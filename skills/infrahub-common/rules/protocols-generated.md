## Protocol Files Are Generated -- Never Edit Directly

Impact: CRITICAL

Infrahub SDK protocol files (typed Python interfaces for
schema nodes) are **generated code** produced by the
`infrahubctl` CLI. They must never be created or edited by
hand. Directly modifying protocol files causes them to drift
from the actual schema, leading to type mismatches, runtime
errors, and silent data corruption.

### How to Identify Protocol Files

Protocol files are typically found in a `generated/`
directory at the project root and contain auto-generated
class definitions that mirror the Infrahub schema. Common
indicators:

- Located in a path like `generated/` or `protocols/`
- Contain a "generated by" or auto-generation comment
  header
- Define Python protocol classes corresponding to schema
  node kinds (e.g., `DcimDevice`, `IpamIPAddress`)
- Import from `infrahub_sdk.protocols`

### Generating Protocols

Use `infrahubctl protocols generate` to create or update
protocol files. The command can work against a **running
Infrahub instance** or directly from **local schema
files**:

```bash
# Generate from a running Infrahub instance (default)
infrahubctl protocols generate

# Generate from local schema directory (no instance needed)
infrahubctl protocols generate --schema-dir schemas/

# Generate from specific schema files
infrahubctl protocols generate \
  --schema-dir schemas/base/ \
  --schema-dir schemas/extensions/
```

Using `--schema-dir` is especially useful during
development when iterating on schema changes without
needing to load them into Infrahub first.

The command generates both **sync** and **async** protocol
classes. Usually only one variant is needed -- choose based
on whether your code uses the sync or async Infrahub SDK
client. In rare cases (e.g., a project mixing sync
utilities with async generators), both may be required.

```bash
# Generate only sync protocols
infrahubctl protocols generate --sync

# Generate only async protocols (default)
infrahubctl protocols generate --async
```

### Correct Workflow

When the schema changes, the correct sequence is:

1. **Update the schema** files (YAML in `schemas/`)
2. **Regenerate protocols**:
   `infrahubctl protocols generate --schema-dir schemas/`
3. **Use the updated protocols** in checks, generators,
   and transforms
4. **Commit** the regenerated protocol files alongside
   the schema changes

### Common Mistakes

| Mistake | Consequence |
| ------- | ----------- |
| Editing protocols to add a field | Overwritten on next regen |
| Editing protocols to fix a type | Masks a schema issue |
| Creating protocol classes by hand | Missing fields, wrong types |
| Forgetting to regen after changes | Stale types in code |

### Prevention

- Treat the `generated/` directory as read-only in all
  development workflows
- After any schema change, always regenerate protocols
  before updating Python code
- Consider adding protocol regeneration to your
  development workflow or CI pipeline
- When reviewing code, flag any diff that includes
  hand-edited protocol files
