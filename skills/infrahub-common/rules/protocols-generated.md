## Protocol Files Are Generated -- Do Not Hand-Edit

Impact: CRITICAL

Protocol files under `generated/` (or `protocols/`)
are output from `infrahubctl protocols generate` —
hand-edits to them are overwritten on the next
regeneration.

### Why it matters

The generator file emits a header comment naming
itself as auto-generated and rewrites the whole file
on each run. A field added by hand to fix a type
error vanishes the next time someone regenerates,
breaking imports in checks and generators that
depended on it. Worse, a hand-edited protocol papers
over a schema-side issue: the type checker passes
locally, then the same code fails at runtime against
a real Infrahub instance because the actual schema
never carried that field. Treat the directory as a
build artifact — fix the schema, regenerate, commit
both together.

### How to Identify Protocol Files

Protocol files are typically found in a `generated/`
directory at the project root and contain
auto-generated class definitions that mirror the
Infrahub schema. Common indicators:

- Located in a path like `generated/` or `protocols/`
- Contain a "generated by" or auto-generation comment
  header
- Define Python protocol classes corresponding to
  schema node kinds (e.g., `DcimDevice`,
  `IpamIPAddress`)
- Import from `infrahub_sdk.protocols`

### Generating Protocols

Use `infrahubctl protocols generate` to create or
update protocol files. The command can work against
a **running Infrahub instance** or directly from
**local schema files**:

```bash
# Generate from a running Infrahub instance (default)
infrahubctl protocols generate

# Generate from local schema directory (no instance needed)
infrahubctl protocols generate --schema-dir schemas/

# Generate from specific schema files
infrahubctl protocols generate \
  --schema-dir schemas/base/ \
  --schema-dir schemas/extensions/
```

Using `--schema-dir` is especially useful during
development when iterating on schema changes without
needing to load them into Infrahub first.

The command generates both **sync** and **async**
protocol classes. Usually only one variant is needed
-- choose based on whether your code uses the sync
or async Infrahub SDK client. In rare cases (e.g., a
project mixing sync utilities with async
generators), both may be required.

```bash
# Generate only sync protocols
infrahubctl protocols generate --sync

# Generate only async protocols (default)
infrahubctl protocols generate --async
```

### Correct Workflow

When the schema changes, the correct sequence is:

1. **Update the schema** files (YAML in `schemas/`)
2. **Regenerate protocols**:
   `infrahubctl protocols generate --schema-dir schemas/`
3. **Use the updated protocols** in checks,
   generators, and transforms
4. **Commit** the regenerated protocol files
   alongside the schema changes

### Common Mistakes

| Mistake | Consequence |
| ------- | ----------- |
| Editing protocols to add a field | Overwritten on next regen |
| Editing protocols to fix a type | Masks a schema issue |
| Creating protocol classes by hand | Missing fields, wrong types |
| Forgetting to regen after changes | Stale types in code |

### Prevention

- Treat the `generated/` directory as a build
  artifact — read-only in normal development
- After any schema change, regenerate protocols
  before updating Python code that consumes them
- Consider adding protocol regeneration to your
  development workflow or CI pipeline
- When reviewing code, flag any diff that includes
  hand-edited protocol files
