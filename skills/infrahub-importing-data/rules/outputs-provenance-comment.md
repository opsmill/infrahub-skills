## Stamp Every Emitted File With a Provenance Comment

Impact: LOW

Each YAML file emitted to `./output_dir/` carries a
top-of-file comment block naming the source CSV(s),
their sha256 fingerprints, the target branch name,
and the emission timestamp. The comment is
informational; the loader ignores it but reviewers
can trace each file back to the input that produced
it.

### Why it matters

Three months after the import, a reviewer looking
at the merged branch sees rows but no record of
where they came from. The lineage rule
([lineage-on-import.md](./lineage-on-import.md))
covers per-value `source` metadata, but that lives
on the Infrahub object. The output YAML file itself
needs a paper-trail too, so re-runs and audits can
match the file against the original CSV without
guesswork.

### Comment shape

The comment is a YAML comment block at the very
start of the file, before the leading `---`:

```yaml
# Generated by infrahub-importing-data
# Sources:
#   inventory.csv (sha256: a3f5...beef, 248 rows)
# Branch: import-data/inventory-20260621-1430
# Emitted: 2026-06-21T14:30:00Z
---
apiVersion: infrahub.app/v1
kind: Object
spec:
  kind: DcimDevice
  data:
    ...
```

Every field on its own commented line so the block
is grep-friendly.

### Correct vs. incorrect

**Correct** — block at the very top, before `---`,
with a `Sources:` line carrying the sha256 and a real
emission timestamp:

```yaml
# Generated by infrahub-importing-data
# Sources:
#   inventory.csv (sha256: a3f5...beef, 248 rows)
# Branch: import-data/inventory-20260621-1430
# Emitted: 2026-06-21T14:30:00Z
---
apiVersion: infrahub.app/v1
kind: Object
spec:
  kind: DcimDevice
  data:
    ...
```

**Incorrect** — comment after `---`, no sha256, and
the emission time reused from the branch name:

```yaml
---
# imported from inventory.csv   # after `---`: some parsers drop pre-data comments here
apiVersion: infrahub.app/v1     # no `Sources:` block and no sha256 — the file isn't traceable
kind: Object                    # reusing the branch timestamp hides a later re-run
spec:
  kind: DcimDevice
  data:
    ...
```

### Multiple input sources

When a file is the result of merging several inputs
(see
[inputs-merge-same-kind.md](./inputs-merge-same-kind.md))
or splitting a denormalized sheet (see
[decomposition-denormalized-csv.md](./decomposition-denormalized-csv.md)),
list each source on its own line:

```yaml
# Generated by infrahub-importing-data
# Sources:
#   manufacturers-na.csv (sha256: 1a2b...c0de, 48 rows)
#   manufacturers-eu.csv (sha256: 3c4d...e0ff, 32 rows)
# Branch: import-data/manufacturers-20260621-1430
# Emitted: 2026-06-21T14:30:00Z
---
apiVersion: infrahub.app/v1
kind: Object
spec:
  kind: OrganizationManufacturer
  data:
    ...
```

The sha256 is the file content's SHA-256 hash, hex-
encoded, truncated to ~8 chars for readability with
a final `...beef`-style ellipsis only if helpful;
the full hash is fine too — readers grep for the
prefix.

### Bootstrap files inherit the same shape

A `00_lineage_accounts.yml` (from
[lineage-on-import.md](./lineage-on-import.md))
that bootstraps an Account for `source` references
gets the same comment, with its source listed as
`(skill bootstrap)`:

```yaml
# Generated by infrahub-importing-data
# Sources:
#   (skill bootstrap — Account for lineage references)
# Branch: import-data/inventory-20260621-1430
# Emitted: 2026-06-21T14:30:00Z
---
apiVersion: infrahub.app/v1
kind: Object
spec:
  kind: CoreAccount
  ...
```

### What the comment is not

- **Not a substitute for value-level lineage.**
  `source` metadata on individual attributes
  (covered by [lineage-on-import.md](./lineage-on-import.md))
  is queryable in Infrahub; the file comment isn't.
  Use both when audit needs both.
- **Not the place for the mapping plan.** The plan
  belongs in the user's confirmation log, not in
  each file. Don't embed column-by-column bindings
  in the comment.

### Common mistakes

- **Putting the comment after `---`.** Some YAML
  parsers handle pre-document comments
  differently. Putting the block at the very top
  (before `---`) keeps it portable.
- **Inventing a timestamp.** Use the actual
  emission time in ISO-8601 with UTC offset (`Z`).
  Don't reuse the branch name's timestamp — the
  emit step can run later if the user re-runs
  after fixing a CSV.
- **Skipping the sha256.** The hash is what makes
  re-runs traceable. A path alone isn't enough —
  the same path may point to different content
  next month.
